##Mars Code

###论文简介

介绍好奇号软件系统中软件可靠性措施。

###论文内容
预防措施主要有三个方面：代码编写标准，被重定义的代码审核流程和逻辑模型检查工具。

####代码编写标准
了解这一领域常出现的错误，列出问题领域清单。

列出仅与风险相关的规则，没有风格相关的规则。

列出规则的两大准则：
+ 规则必须与早期观察的风险相关；
+ 能够基于工具的检查方法验证。

如何遵循规则（四级合规）：
+ LOC1 所有代码的最低标准。所有代码都可通过编译器的良好静态源代码分析器的考验。
+ LOC2 确保代码的可预测执行。
+ LOC3 降低残留缺陷，使用大量断言。
+ LOC4 将限制对 C 预处 理程序、函数指针和指针间接寻址的使用。
+ LOC5&LOC6 天假MISRA C代码中的所有规则

还有一项预防措施，就是对软件开发人员的认证。

####基于工具的代码审核
采用静态源代码分析工具，并且对所有代码运行四个分析器以提高检测出的错误数量。和同行代码审核相结合，代码模块所有者需要对这两者的反馈都进行相应，并作出自己的回复。

####模型检查
使用Spin模型检查工具，能够快速有效的对异步线程执行进行验证。
然后以一个对共享内存的存储的读和写的并行操作来示例自动化测试工具。


###学习到的知识
要实现一个安全可靠的软件系统需要：指定代码规则，符合规则的代码编写，代码的多次审核，其中充分利用现有的代码审核和模型检查工具。

##The Scalable Commutativity Rule : Designing Scalable Software for Multicore Processors

###论文简介
设计了一种规则-commutativity rule-用于设计可拓展程序。
提供了一种可以通过接口判断程序可拓展性的方法。
设计一个工具，能够自动检测程序间共享交互的冲突。

###论文内容
首先将系统中的各个调用差分成小粒度的action，而后以此为基础定义了SI-commutes，用来指定一串action是否和顺序有影响。

而后基于这个定义，提出了自己的rule，指出能满足SI-commutes，并且程序能够正确部署的话，那样对于一系列action就存在一种实现能够conflict-free。并给与了证明。

在Sec4中介绍了设计可交互的接口的一些原则：
+ 操作分解；
+ 不强制确定结果
+ 在可允许的时候不确定操作顺序
+ 异步释放资源

并且设计了COMMUTER工具用于分析现有系统的接口间的可拓展性。


###论文贡献点
形式化了接口的可拓展性，并且提出了可拓展性的设计建议。

设计了工具用于分析现有系统的拓展性。

###论文缺点
给出了设计建议，都可以支持并行，但没有考虑多带来的开销，例如将操作细化会增加程序总量，并且增加开发难度。
